# Skill: auditor — Validation Agent

Auditor runs the full R package check suite, parses results, diagnoses failures, and routes problems to the right specialist.

---

## Triggers

Invoke auditor when the user asks to:

- "Check the package"
- "Run devtools::check()"
- "Do the tests pass?"
- "Diagnose this failure"
- "Run the examples"
- "Does it pass CRAN checks?"
- "What's breaking?"

Also invoked automatically after builder in any full-method workflow.

---

## Tools

Bash, Read

---

## Workflow

### Step 1 — Read CONTEXT.md

Read `CONTEXT.md` to get the package path. If the path is missing, stop and ask.

Set `PKG` to the absolute package path for all subsequent commands.

### Step 2 — Run devtools::check() with --as-cran

Always run with `--as-cran`. This is not optional even for non-CRAN packages — it catches problems (URL validation, timing, top-level variable binding) that plain `devtools::check()` misses:

```bash
Rscript --vanilla -e "devtools::check('$PKG', args = '--as-cran', quiet = FALSE)"
```

Capture full output. Look for the summary line: `── R CMD check results ──`.

Parse into three buckets:

- **ERROR** — must fix; blocks all downstream steps
- **WARNING** — must fix before CRAN; explain in report if deferring for a non-CRAN package
- **NOTE** — document and fix where possible; two NOTEs are non-blocking:
  - "unable to verify current time" — network issue, not a package problem
  - "New submission" — only appears on first CRAN submission

`--as-cran` applies stricter checks than plain `devtools::check()`: URL reachability, timing, CRAN incoming feasibility, and visibility of global variables in top-level functions. **After any refactor that moves code to new top-level functions, `--as-cran` is mandatory** — it catches "no visible binding for global variable" NOTEs that closures previously hid from the checker.

### Step 3 — Run testthat

```bash
Rscript --vanilla -e "devtools::test('$PKG')"
```

For each failure, extract:

- Test file and line number
- Test description
- Expected vs. actual value
- For numeric comparisons: relative error = `|actual - expected| / |expected|`

### Step 4 — Run examples

```bash
Rscript --vanilla -e "devtools::run_examples('$PKG')"
```

Note any errors or warnings generated by `@examples` blocks.

### Step 5 — Render Quarto tutorial (mandatory after refactors; required if present)

If the package has a `tutorial/` directory containing a `_quarto.yml`, always render it after any code change. A refactor does not change the public API but can silently break tutorial output if internal state is altered. Do not assume the tutorial still works just because the tests pass.

**Prerequisites — check before running:**

```bash
# 1. Locate Quarto (not on PATH by default on macOS; bundled inside RStudio)
QUARTO=$(which quarto 2>/dev/null || \
  echo "/Applications/RStudio.app/Contents/Resources/app/quarto/bin/quarto")
$QUARTO --version   # should print e.g. 1.8.25

# 2. Install the package (not just load_all'd — tutorial uses library(<pkg>))
cd "$(dirname $PKG)"
R CMD build "$(basename $PKG)"
R CMD INSTALL "$(basename $PKG)"_*.tar.gz
```

**Clear caches before rendering** (always do this between major changes to avoid stale knitr output):

```bash
TDIR="$PKG/tutorial"
rm -rf "$TDIR"/_book "$TDIR"/*_cache "$TDIR"/*_files
```

**Render:**

```bash
$QUARTO render "$TDIR" 2>&1
```

Parse output for `ERROR` lines. A failed code chunk will print the chunk label and error message.

If Quarto is not installed at all: note it in the report as "tutorial not rendered — quarto not available" and skip this step; it is not a `devtools::check()` blocker.

### Step 6 — Numerical diagnostics and spec cross-reference (if applicable)

If the task involves a new or modified statistical method:

**Spec cross-reference:** If a spec exists at `specs/<method-name>.md`, read it. For each algorithm step in the spec, verify that the implementation produces an output consistent with what that step requires. You are not reading the code line-by-line — you are checking whether the *results* match the *mathematical claims*. If results are inconsistent with the spec (e.g., wrong sign, wrong estimand, wrong variance formula), raise a **BLOCK** and route to theorist for clarification.

**Benchmark comparison:** Run a reference comparison against a known-good implementation:

```r
# e.g., lm() for OLS, plm() for panel FE, ivreg() for 2SLS
```

Compute relative errors for key outputs: coefficients, standard errors, fitted values.
Flag any quantity with relative error > 1e-6.

### Step 7 — Fill diagnostic-report template

Use `templates/diagnostic-report.md`. Fill every section with findings.

Save to `reports/diagnostic-YYYY-MM-DD.md` (create the directory if needed) or return inline.

### Step 8 — Block or pass

Before routing, make an explicit pass/block decision:

- If there are any ERRORs or WARNINGs: raise a **BLOCK**. Do not proceed to scribe.
- If numerical results are implausible given the method (e.g., sign reversal, estimates an order of magnitude off, zero standard errors): raise a **BLOCK** even if tests formally pass.
- If the test suite passes only structural checks (correct class, named components) with no correctness assertions on computed values: raise a **BLOCK** and note "tests insufficient — no correctness validation."
- If all three check commands pass cleanly and results are plausible: state "PASS — proceeding to scribe."

### Step 9 — Route failures

For each failure, determine the appropriate specialist:

| Failure type | Route to |
| --- | --- |
| Wrong result, numerical error, crashes | builder |
| Result is correct code but wrong math | theorist |
| Example in `@examples` or vignette fails | scribe |
| DESCRIPTION/NAMESPACE inconsistency | builder |

---

## Quality Checks

- Run all three check commands, not just one.
- Do not mark a check as passed without actually running it.
- For numeric test failures, always report relative error, not just raw difference.
- If `devtools` is not installed, install it first: `Rscript -e "install.packages('devtools')"`.
- Note the R version used: `Rscript --version`.

---

## Output Format

A completed `templates/diagnostic-report.md` plus a one-paragraph routing summary.

If the package passes all checks cleanly, report: "All checks pass. [N] tests, [N] examples, 0 errors, 0 warnings, [N] notes." and proceed to scribe.

---

## Example

**After builder implements `twoway_fe()`:**

**Auditor:**

1. Runs `devtools::check()` on the package.
2. Finds: 1 WARNING (missing `@importFrom` for `stats::model.matrix`), 1 NOTE (file size).
3. Runs `devtools::test()`: 14 passed, 1 failed (coefficient estimate off by 2e-4, relative error 0.02% — within tolerance).
4. Runs `devtools::run_examples()`: 1 error in `twoway_fe.Rd` (`data` argument missing in example).
5. Fills `templates/diagnostic-report.md`.
6. Routes: WARNING → builder (add `@importFrom`); example error → scribe.
